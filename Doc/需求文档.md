# NewLife.Audio 需求文档

> 版本：v1.0 | 日期：2026-06-29 | [← 返回 README](../Readme.MD) | [English README](../Readme.en.MD)

本文档描述 NewLife.Audio 的愿景、核心目标和模块划分。完成状态在[功能模块清单](功能模块清单.md)中追踪，详细设计在各模块架构文档中展开。

---

## 1. 背景与愿景

### 1.1 系统定位

NewLife.Audio 是一个**跨平台 .NET 全功能音频处理库**，面向 .NET 开发者提供编解码、设备 I/O、DSP 效果、语音处理、容器格式读写、流媒体传输等一站式音频能力。纯托管 C# 优先，NuGet 一键安装即用，无原生依赖门槛。

### 1.2 愿景

- **跨平台统一体验**：一份代码跑通 Windows / Linux / macOS，开发者无需关心底层音频后端差异
- **纯托管优先**：常用算法全部 C# 实现，降低部署成本；仅对性能/质量敏感场景（AAC/Opus/AEC）可选包装原生库
- **IoT 深度兼容**：延续海思芯片头处理基因，扩展大华/宇视等厂商适配，服务物联网音视频场景
- **信号链可组合**：借鉴业界成熟的信号链模式，让编解码、DSP、语音处理模块像搭积木一样自由组合。

## 2. M1 编解码引擎

音频编解码是库的核心能力，提供 PCM ↔ 各种编码格式的双向转换。

- **M1-1 PCM ↔ G.711 μ-law 编码**：补齐现有 `G711UCodec.FromPcm` 的 `NotImplementedException`，实现完整的双向 G.711 μ-law 编解码。
- **M1-2 PCM ↔ G.711 A-law 编解码**：已有完整实现，维持现有 `G711ACodec`。
- **M1-3 PCM ↔ IMA ADPCM 编解码**：已有完整实现，维持现有 `ADPCMCodec`。
- **M1-4 PCM ↔ AAC 编解码**：将 `FaacEncoder` 接入 `AudioCodecFactory` 统一管理；新增 AAC 解码能力（可选包装 FDK-AAC 或纯托管实现）。
- **M1-5 PCM ↔ Opus 编解码**：包装 libopus 原生库，提供高质量、低延迟的互联网音频编解码，覆盖 6kbps~510kbps 全码率。
- **M1-6 PCM ↔ MP3 编解码**：解码端优先纯 C# 实现；编码端可选包装原生 MP3 编码库或纯托管实现。
- **M1-7 PCM ↔ FLAC 编解码**：无损音频压缩，优先纯 C# 实现编解码两端。
- **M1-8 PCM ↔ Ogg Vorbis 编解码**：优先纯 C# 实现，为 Opus 提供 Ogg 容器基础。
- **M1-9 PCM ↔ G.722 / G.726 编解码**：电信语音编解码器，纯 C# 实现，服务 VoIP/对讲场景。
- **M1-10 PCM ↔ Speex 编解码**：语音编解码器，远期可选，视需求决定。
- **M1-11 编解码工厂插件化**：改造 `AudioCodecFactory`，支持运行时注册/发现编解码器，根据 `AVTypes` 或媒体类型自动路由。
- **M1-12 芯片音频头适配**：扩展海思头处理（已有），增加大华、宇视等主流 IoT 芯片厂商的音频头识别与处理。

## 3. M2 音频设备 I/O

提供播放、录制、设备管理等音频硬件交互能力，提供完整的设备抽象层。

- **M2-1 设备枚举与发现**：枚举系统音频输入/输出设备，获取名称、通道数、支持格式等元数据。
- **M2-2 WaveOut 播放**：Windows WaveOut API 封装，提供低延迟音频播放，作为 Windows 平台最简单可用的后端。
- **M2-3 WASAPI 播放与录制**：Windows WASAPI（共享/独占模式）封装，支持低延迟播放、麦克风采集和环路录制。
- **M2-4 ASIO 低延迟播放与录制**：专业音频 ASIO 驱动封装，面向录音/监听等需要极低延迟的场景。
- **M2-5 跨平台后端抽象**：Linux（ALSA/PulseAudio）和 macOS（CoreAudio）音频后端，统一 `IAudioPlayer`/`IAudioRecorder` 接口。
- **M2-6 音频采集**：麦克风录音，支持回调模式和阻塞读取模式。
- **M2-7 环路录制**：系统音频输出回采（WASAPI Loopback / PulseAudio Monitor），用于屏幕录制、直播推流等场景。
- **M2-8 多设备并发管理**：同时管理多个播放/录制设备，互不干扰。

---

## 4. M3 音频处理（DSP）

纯托管 C# 实现的音频数字信号处理管线，借鉴业界成熟的信号链组合模式。

- **M3-1 重采样**：支持线性插值、Sinc 插值等多种算法，任意采样率互转。
- **M3-2 声道转换**：单声道↔立体声↔多声道互转，支持自定义声道映射矩阵。
- **M3-3 位深度与采样格式转换**：8/16/24/32 bit 整数与 32 bit 浮点数之间的自由转换。
- **M3-4 音量控制与增益**：线性/对数增益调节，支持静音检测。
- **M3-5 混音引擎**：多路音频流按权重混合为一路，支持 32-bit 浮点精度累加后归一化。
- **M3-6 FFT 频谱分析**：快速傅里叶变换，输出频谱数据用于可视化或音频特征提取。
- **M3-7 均衡器**：多段参数均衡器，支持低架/峰值/高架滤波器的组合配置。
- **M3-8 BiQuad 滤波器**：低通/高通/带通/带阻/峰值/低架/高架等标准滤波器。
- **M3-9 动态范围压缩与扩展**：压缩器/限幅器/噪声门，用于响度控制和动态优化。
- **M3-10 淡入淡出与交叉淡变**：音频片段平滑过渡，支持线性/对数/余弦等多种曲线。
- **M3-11 变速与变调**：改变播放速度或音高，且两者可独立控制。
- **M3-12 信号链管线**：`IAudioPipeline` 将多个 DSP 处理器串联为处理链，提供流畅的组合式 API。

## 5. M4 语音处理

实时语音通信所需的前端处理能力，填补 .NET 生态空白。已全部实现。

- **M4-1 语音活动检测（VAD）**：检测音频流中的人声片段，纯 C# 实现业界标准的 GMM 模型 VAD 算法。
- **M4-2 声学回声消除（AEC）**：消除扬声器到麦克风的回声，保证全双工通话质量。纯托管 NLMS 自适应滤波器 + 双讲检测 + 舒适噪声注入，已实现。
- **M4-3 自动增益控制（AGC）**：自动调节麦克风或输入音频的增益，保持恒定输出电平。
- **M4-4 噪声抑制（NS）**：抑制稳态背景噪声（风扇、空调等），提升语音清晰度。基础算法纯 C#，高质量方案远期引入 ML 模型。
- **M4-5 高通滤波**：去除低频直流偏移和次声频噪声。
- **M4-6 语音前置处理管线**：将 VAD → AGC → NS → 高通滤波组合为即开即用的语音前端处理链。

## 6. M5 容器格式

音频容器（文件格式）的读写支持，作为编解码数据的持久化入口。

- **M5-1 WAV 文件读写**：完整 WAV/PCM 文件的读取（含 RIFF 头解析）和写入（已有部分 `ToWav` 辅助方法），支持 PCM/ADPCM/A-law/μ-law 等编码标记。
- **M5-2 Ogg 容器封装与解封**：纯 C# Ogg 容器读写，为 Vorbis 和 Opus 提供容器基础。
- **M5-3 MP4/M4A 容器**：读取和写入 AAC/MP3 音频的 MP4 容器。
- **M5-4 FLAC 原生容器**：FLAC 格式自身的容器读写（含元数据块、SEEKTABLE 等）。
- **M5-5 原始 PCM 头解析**：无头 PCM 数据的格式自动推断（采样率、位深、声道数）和手动指定。

---

## 7. M6 流媒体与传输

音频网络传输能力，服务实时对讲、直播推流、IoT 设备上报等场景。已全部实现。

- **M6-1 RTP 音频包封装与解析**：按 RFC 标准将音频帧封包为 RTP 包，支持 G.711/Opus/AAC 等 Payload Type。
- **M6-2 RTSP 音频流**：RTSP 协议客户端，拉取或推送音频流。
- **M6-3 HTTP 音频流**：HTTP 实时音频流（Shoutcast/Icecast 风格），支持分段传输和流式播放。
- **M6-4 WebSocket 音频推送**：基于 WebSocket 的双向低延迟音频传输，服务 Web 端实时通话。

## 8. 🧊 暂缓清单

计划做但现在不做，注明明确理由和可重新评估的前提。

| 功能 | 暂缓理由 | 解冻前提 |
|---|---|---|
| Speex 编解码（M1-10） | Opus 已全面替代 Speex，实际需求极少 | 有明确 IoT 设备仅支持 Speex 的场景出现 |
| 噪声抑制高质量版（M4-4 远期） | 深度学习 NS 需引入 ONNX Runtime 等重量级依赖 | 用户明确需要且愿意接受 ONNX 依赖 |
| MIDI 支持 | 专用领域，非通用音频库核心 | 出现明确且广泛的 MIDI 处理需求 |
| 3D 空间音频 | 专用于游戏领域 | NewLife 游戏框架立项或有空间音频通用需求 |

## 9. 不做什么

明确排除的功能项，避免后续争议。

- **视频编解码**：属于 `NewLife.Video` 或独立视频库的范畴，许音频库介入会模糊边界、膨胀依赖。
- **语音识别（ASR）与语音合成（TTS）**：独立领域，应使用专用语音识别库。NewLife.Audio 提供原始音频数据，由上层 ASR/TTS 库消费。
- **深度学习模型训练与推理**：不内置任何 ML 模型，不依赖 ONNX Runtime / PyTorch。如需 ML 能力，由调用方自行集成。
- **音乐制作/DAW 功能**：VST 插件宿主、MIDI 编排、多轨编辑等属于专业音乐制作领域，不在通用音频库范畴。
